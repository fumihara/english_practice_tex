\section{はじめに}
分析によって，正確で完全で一貫した明白なシステムのモデルが得られる．
たとえどんな誤りや曖昧な部分があっても，
開発者は要求獲得で提示された要求仕様を形式化し，
より詳細な境界条件や例外事例を調査する．
要求仕様を変更しなければならない場合や追加の情報を集めなれけばならない場合に，
顧客やユーザはたいていこの作業に巻き込まれる．

オブジェクト指向分析では，開発者はアプリケーションドメインを記述するモデルを構築する．
例えば，時計の分析モデルは時計がどのように現在時刻を表すか，

分析モデルはそのような場合，どのようにアクタやシステムが相互作用して
アプリケーションドメインモデルを操作するかを記述するために拡張される．
例えば，時計の所有者がどのように時間や曜日をリセットするのかなど．
開発者は，上位レベルの設計中に%(6章)
開発されるシステムの構成に備えて，
非機能要件とともに分析モデルを使用する．

\subsubsection{分析モデルの再調査}%5.4.11節 %review 再調査
分析モデルは段階的に反復して構築され，
最初の時点では正しいことも，完成していることさえめったにない．
開発者によって設計や実装のための
分析モデルが使用可能な正しい仕様にまとまる前には，
顧客やユーザとの数回の反復が必要である．
%例とか

一旦モデルに対する変更の数が最小限となり，また変更の範囲が
ローカライズされていれば，
分析モデルは安定となる．
それから，
まずは開発者によって，その後は開発者と顧客の共同で，
分析モデルは見直される．
見直しの目的は，要求仕様が正しく完全で一貫性があり曖昧でないと確認することである．
さらに，開発者と顧客は要求が現実的で検証可能かどうかも見直す．
開発者は下流でエラーを発見し
仕様を変更することに備えるべきであるが，
可能な限り要件のエラーを上流で
見つけるのは価値ある投資である．
見直しをすることでチェックリストや質問事項を容易にすることができる．
以下は
[Jacobson，1999]や[Rumbaugh，1991]に採用された質問例である．

次の質問は，モデルが正しいことを保証するために
されなければならない．
\begin{itemize}
 \item エンティティオブジェクトの用語集はユーザが理解しやすいか？
 \item 抽象クラスはユーザレベルの概念に対応しているか？
 \item すべての記述はユーザの定義に従っているか？
 \item すべてのエンティティオブジェクトとバウンダリオブジェクトは
 名前として意味のある名詞句を持っているか？
\item 全てのエラーが記述され処理されているか？
\end{itemize}

次の質問は，モデルが完全か保証するためにされるべきである．
\begin{itemize}
 \item 各オブジェクトに対して:任意のユースケースで必要か？
 そのユースケースは作成され，修正され，破棄されているか？
 バウンダリオブジェクトからアクセスできるか？
 \item 各属性に対して:いつ設定されるか？ 型は何か？ 
 修飾子であるべきか？ 
 \item 各関連に対して:いつ横断するか？%横断？？？？ 
 なぜ特定の多重度が選ばれたのか？%特定？
 一対多や多対多の多重度は修飾できるか？
 \end{itemize}

 次の質問はモデルの整合性を確保するためのものである．
 \begin{itemize}
 \item 同じ名前の複数のクラスやユースケースはあるか？
 \item 似た名前のエンティティは似た概念を表しているか？
 \item 同じ汎化階層にないが似た属性や関連を持つオブジェクトはあるか？
 \end{itemize}

次の質問は分析モデルによって記述されたシステムが現実的か
確認するためのものである．
 \begin{itemize}
 \item システム内に新しい特徴はあるか？その実現可能性を
 確かめるために構築された研究やプロトタイプはあるか？
 \item 性能と信頼性は要求を満たすか？ これらの要求は
 選択したハードウェアで実行されているプロトタイプ
 で検証されたか？
 \end{itemize}

\subsubsection{責任の割り当て}%5.5.2節
分析には個々の広範囲の関与を必要とする．%個体？
ターゲットのユーザはアプリケーションドメインの知識を
提供する．
顧客はプロジェクトの資金を提供し
ユーザ側の労力を調整する．
アナリストはアプリケーションドメインの知識を引き出し
それを形式化する．
開発者は実現可能性とコストのフィードバックを提供する．
大規模なシステムでは，
多くのユーザやアナリスト，
開発者が関与し，
プロジェクト要件の統合，通信時に追加の課題を導入する．
この課題は，明確に定義された役割と範囲を個々に
割り当てることで満たすことができる．
役割には主なタイプとして
情報の発生，統合，見直しの3つがある．
\begin{itemize}
 \item エンドユーザは現在のシステムや将来のシステム環境，
 サポートするべきタスクに関する情報を生成する
 アプリケーションドメインの専門家である．
 各ユーザは1つ以上のアクタに対応し，関連するユースケース
 を識別するのに役立つ．
 \item インテグレーションの役割である顧客は，
 ユーザの要求に基づいてシステムの範囲を定義する．
 ユーザごとにシステムの視点は違い，
 彼らはシステムの様々な部分から利益を得るか
 将来のシステムについての様々な意見や期待を持っているかの
 どちらかである．顧客はアプリケーションドメインの情報
 を統合しユーザの期待を元に矛盾を解決する．
 \item アナリストは現在のシステムをモデル化し将来のシステム
 について情報を生成するアプリケーションドメインの専門家である．各アナリストは最初に1つ以上のユースケースの詳細化を
 担当する．%ちょっと続く
 \item インテグレーションの役割である設計者はシステムの観点から
 ユースケースやオブジェクトモデルを統合する．%アナリストと云々
 \item ドキュメントの編集者はドキュメントの低レベルな統合や，

 \item 構成管理者は他のドキュメントとのRADに関する追跡可能な情報だけでなく， ドキュメントの改訂履歴の保持なども担当する(6章を参照)．
 \item レビュアはRADの正確さ，完全性，一貫性，明確さを検証する．要件の検証中にはユーザや顧客，開発者などもレビュアになる可能性がある．

\subsubsection{分析モデルを超えた反復}%5.5.4節 ？？？？？？？？
分析では，
しばしばシステム設計や実装など他の開発作業と並行して，

%なんか

要求作業は，安定したモデルに向かって収束していく
いくつかの段階として見ることができる．
\begin{itemize}
 \item ブレインストーミング\\
 他の開発作業が始まる前は，要求はブレインストーミングの

 \item 固定\\
 一度顧客と開発者が共通の考えに収束し，
 システムの境界を定義して標準的な用語のセットに同意すれば，
 固定を開始する．
 \item 成熟\\
 より高いレベルでの変化は依然として可能だが困難になるため，
 更に慎重に実行される．
 \end{itemize}
 %色々
 
顧客が要件に同意してからは，
分析モデルへの変更は記述漏れやエラーに対処するべきである．
特定の設計チームの開発者は，
モデルの整合性が損なわれないように確認する必要がある．
要求モデルは構成管理下にあり，変更は既存の設計モデルに
伝播させるべきである．
 
特徴やシステムの機能の数は時間とともに常に増加し，
それぞれの変更はシステムの整合性を脅かす可能性がある．
新しい特徴や新機能をシステムに追加する場合は，
顧客に要求されたものかどうか，既存の機能への影響はどうかなどの質問とともに挑戦するべきである．
 
 変更が必要なときは，顧客や開発者は
 変更の範囲や要求された結果を定義し，
 分析モデルを変更する．
 システムには完全な分析モデルが存在することを考えると，
 新機能を明記するのはより簡単である(実装はより難しい)．

再び現れる．

プロジェクトにおいて重要な節目である．
要求分析のドキュメントの受け入れは，
多くの作業が分析モデルに依存することを考えると，
他のドキュメントよりも重要である．
